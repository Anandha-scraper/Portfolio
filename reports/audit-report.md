# Portfolio Audit — "The Living Blueprint"

**Scope:** read-only static analysis. No code was changed in this pass.
**Stack (verified from `package.json` / config):** Next.js `16.2.9` (App Router only — no `pages/` dir exists), React `19.2.7`, TypeScript `5.9.3` (`strict: true`), Tailwind v4 (no `tailwind.config.*` — theme lives in `@theme` blocks in `app/globals.css`), `output: "export"` static export deployed to Firebase Hosting. No ESLint config or dependency exists anywhere in the repo (`"lint": "npm run typecheck"` in `package.json:10`).
**Build verified:** `npm run typecheck` — 0 errors. `npm run build` — succeeds, static export to `./out` (14 MB total).
**Note on repo state:** at audit time the working tree had 6 uncommitted, in-progress local edits (`mission-control.tsx/css`, `name-patrol-sprite.tsx`, `master.tsx`, `project-preview-frame.tsx/css`). All findings below, and the build numbers, reflect that exact state, not a clean `origin/main` checkout.

## 3. Top 10 Findings (ranked)

1. **Broken social share card.** `openGraph`/`twitter` metadata (`app/layout.tsx:78-92`) has no `images`, yet `twitter.card` is explicitly `"summary_large_image"` — a card type that *requires* an image slot. Every link share of the portfolio (LinkedIn, Twitter/X, Slack, Discord) will render broken or image-less at exactly the moment it matters most.
2. **1.36 MB unoptimized hero image at the LCP position.** `public/profile/anandha.png` is 1536×1024, loaded via `next/image` with `priority` + `fill` (`components/mission-control/mission-control.tsx:47-58`) — but `next.config.ts:8-10` sets `images.unoptimized: true` (required under `output: "export"`), so none of `next/image`'s resize/format benefits actually apply. This ships at full weight to every device.
3. **Second ~1 MB unoptimized image, not even going through `next/image`.** `public/project-preview.png` (990 KB, 1329×1044) is a CSS `background-image` (`components/project-ecosystem/project-preview-frame.css:21`), loaded unconditionally in the first scroll section.
4. **Confirmed WCAG AA contrast failure**, computed from the actual token values in `app/globals.css:19` (`--color-ink-faint: #6f6651`) against `--color-canvas`/`--color-surface`: **3.31:1** and **3.04:1** respectively, applied to real 12px text (`components/blueprint/section-heading.css:21-25`, `.section-heading__index`). WCAG AA requires 4.5:1 at this size. Not a guess — computed via the standard relative-luminance formula from the exact hex values in the CSS.
5. **Heading hierarchy skips h2 for an entire section.** The DOM goes h1 (`components/mission-control/mission-control.tsx:79`) straight to a conditional h3 (`components/project-ecosystem/project-dungeon-panel.tsx:60`, only rendered when a project is flagged `updating`) — `project-ecosystem.tsx` has zero headings of its own for the whole Project Ecosystem section.
6. **Inconsistent code-splitting.** `ProjectEcosystem` (the canvas/dungeon-map system) is statically imported (`app/page.tsx:4`) while `CapabilityNetwork` and `ContactStudio` — equally below-the-fold, defined two lines later in the same file — are correctly `next/dynamic`-imported (`app/page.tsx:7-14`).
7. **No favicon anywhere in the repository** — confirmed absent from both `app/` and `public/` by directory search. A one-page portfolio's browser tab currently shows a generic icon for every visitor.
8. **`lib/sprite-control.ts`'s "master control" claim is false in practice.** Its own doc comment (`:1-11`) says components read sprite metadata from there "instead of hard-coding magic numbers," but 5 separate files hardcode their own copies of the same data instead of importing it: `components/book/magic-book.tsx:18-23`, `components/ui/asset-gallery.tsx:52-284`, `components/capability-network/island-node.tsx:76`, `components/project-ecosystem/dungeon-treasures.tsx:45`, `components/capability-network/capability-network.tsx:403`.
9. **Animation loops with no visibility gating at all.** `components/companion/skeleton-companion.tsx` and `components/mission-control/name-patrol-sprite.tsx:81-109` both run `requestAnimationFrame` loops that never check scroll/viewport visibility — confirmed via direct inspection of both files' effect bodies (only `reduceMotion` gates them). Meanwhile the codebase has 3 *different* competing implementations of exactly this "pause offscreen" pattern (`hooks/use-in-viewport.ts`, `components/ui/pixel-sprite.tsx:31-60`, `components/ui/particles.tsx:276-280`) that these two components could have used instead.
10. **No `robots.txt`, sitemap, or web manifest anywhere**, confirmed absent by full-repo search — despite `metadata.robots = { index: true, follow: true }` being explicitly set (`app/layout.tsx:93`), there's no actual `robots.txt` backing that intent, and no sitemap for the (admittedly single-page) route to be discovered by.

**Sections with zero issues found:** `dangerouslySetInnerHTML` usage — none anywhere in the repo. Exposed secrets/env vars — none; no `.env*` file exists, no `process.env` reference exists anywhere in first-party code. `role="button"`-on-a-real-`<button>` redundancy — none found. Package-level dead dependencies — every entry in `package.json` is genuinely used somewhere (see full cross-check in the architecture research; the only soft note is `react-dom` having no *direct* import statement anywhere, which is normal/expected for a Next.js app).

## 4. Unverified — needs manual check

- **Real LCP/CLS/INP/TTFB numbers.** No Lighthouse or equivalent run was performed this pass (would require serving the built `out/` and driving a real or headless browser). Everything performance-related above is grounded in verified file sizes, image dimensions, bundle composition, and code-splitting decisions instead — real signals, but not a substitute for an actual trace.
- **`role="application"` + `tabIndex={0}` + window-level `keydown` capture on the dungeon map** (`components/project-ecosystem/dungeon-map.tsx:353-354,480-486`). This is a legitimate pattern for a real WASD-driven canvas game, but whether it behaves acceptably for assistive-tech users tabbing onto it (focus handoff, whether the terminal `<input>` in the Contact section ever fights with the window-level key listener) needs a real screen reader + keyboard pass, not static reading.
- **Actual runtime re-render behavior / missing memoization.** Only 10 of 34 component files use `useMemo`/`useCallback`, and `React.memo` is used nowhere in the tree. No React DevTools Profiler trace was run, so I can't assert any *specific* component re-renders expensively in practice — flagging the low memoization adoption as a fact, not a confirmed bug, per the audit's own instruction not to recommend memoization without profiling evidence.
- **Screen-reader announcement behavior** for the `aria-hidden`/decorative-image pairings throughout (all read as correct from markup, but not tested with an actual AT).
